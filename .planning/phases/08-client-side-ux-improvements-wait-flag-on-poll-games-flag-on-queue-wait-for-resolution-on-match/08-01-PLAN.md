1. In `src/main.rs`, update the `Poll` variant to add three optional flags:
   - `--wait` (bool flag, default false): enables polling loop
   - `--wait-interval` (u64, default 3): seconds between polls
   - `--wait-timeout` (u64, default 300): max seconds before giving up

   ```rust
   Poll {
       /// Wait until match has status waiting_submissions with a problem
       #[arg(long)]
       wait: bool,
       /// Polling interval in seconds (default: 3)
       #[arg(long, default_value = "3")]
       wait_interval: u64,
       /// Maximum wait time in seconds (default: 300)
       #[arg(long, default_value = "300")]
       wait_timeout: u64,
   },
   ```

2. Update the `Commands::Poll` match arm in `run()` to pass the new args:
   ```rust
   Commands::Poll { wait, wait_interval, wait_timeout } => {
       let client = HttpClient::new(&backend_url, signer, address, &private_key_hex)?;
       commands::poll::execute(&client, &address, fmt, wait, wait_interval, wait_timeout).await
   }
   ```

3. In `src/commands/poll.rs`, update the `execute` function signature to accept `wait: bool, interval_secs: u64, timeout_secs: u64`.

4. When `wait` is false, behavior is identical to current (no change).

5. When `wait` is true, implement a polling loop:
   - Record start time using `std::time::Instant::now()`
   - Loop:
     a. Call the existing poll logic (GET `/matches/active/{address}`, handle ready flow)
     b. Check if the response has `match.status == "waiting_submissions"` AND `match.problem` is non-null
     c. If yes: print final result and return
     d. If elapsed > timeout_secs: print timeout message and return the last response
     e. If table mode: print a short status line like `"[12s] Polling... status: waiting_ready"` (overwrite previous line with `\r` or just print new lines)
     f. Sleep `interval_secs` seconds via `tokio::time::sleep`
     g. Continue loop

6. Extract the single-poll logic into a helper function `poll_once(client, address) -> Result<serde_json::Value>` that does the GET + ready acknowledgement + waiting_start handling and returns the final JSON data. The existing `execute` logic (without the final print) becomes this helper. Both the wait loop and non-wait path call `poll_once` then `print_poll_result`.

7. In table mode during wait loop, print progress like:
   ```
   [  3s] Polling... no active match
   [  6s] Polling... status: waiting_ready
   [  9s] Polling... status: waiting_submissions (problem ready!)
   ```
   In JSON mode, suppress intermediate output -- only print the final result.
